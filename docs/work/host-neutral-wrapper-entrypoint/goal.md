# Work Goal: Invoke one tailored CLI through its ordinary command name

- Status: Active
- Retention: temporary
- Retention reason: None
- Governing contract: ADR 0008 and the wrapper-materialization sections of
  `docs/00_theses.md` through `docs/04_harness.md`
- Review/delete trigger: Delete after durable conclusions are promoted and the change completes
- Successor: None
- Owner: Codex
- Target: First host-neutral wrapper entry-point iteration
- Related ADRs: 0005, 0006, 0007, and 0008

## Outcome

A maintainer with one exact adopted bundle can ask `atr` for a deterministic
POSIX-shell wrapper function, activate that function in a caller-owned shell,
and invoke the ordinary source-command spelling. The function forwards exact
argv to a host-neutral Atsura wrapper entry point, which reconstructs the same
fresh plan as the direct gateway, invokes the exact physical source at most
once, and emits only the plan-declared tailored result.

The wrapper knows nothing about coding-agent hosts, hooks, models, sessions,
permissions, or host settings. Downstream integrations may call the same
generated function, but they own their activation and compatibility evidence.

## Why now

The completed runtime proves deterministic compilation, adoption, preview, and
one typed JSON transformation, but a user still has to invoke `atr bundle
execute` and receives a maintainer evidence envelope. That does not yet prove
the defining experience of using the ordinary tailored CLI command.

An attempted host-adapter design exposed the wrong layer: it required Atsura to
parse and rewrite vendor tool requests. The product owner clarified that the
agent invokes an already rewritten function and that agent-specific concerns
do not belong in Atsura or an Atsura adapter. ADR 0008 records that correction.
The retired deferred capability evidence is recorded in
[capability-retirement.md](capability-retirement.md).

## Non-goals

- Claude Code, Codex, or other coding-agent adapters or lifecycle commands.
- Editing or attesting shell startup files, agent settings, hooks, trust,
  permissions, or sandbox rules.
- Executable/PATH shim installation in this first slice.
- General shell generation, configuration-authored scripts, `eval` inside
  Atsura, or `sh -c` source execution.
- Multiple purpose profiles, source refresh, raw execution, identity-wrapper
  execution, argv-only execution, RTK runtime, or a second source adapter.
- Claiming that missing external activation prevents direct source invocation.
- Coding-agent-host fixtures or compatibility claims inside Atsura.
- Publication, tag, package-manager release, or release creation.

## Acceptance criteria

- [ ] Exact scoped help lets a maintainer discover the wrapper-render and
      wrapper-run outcomes and all required inputs without source inspection or
      an undeclared parser.
- [ ] One exact adopted bundle produces byte-stable POSIX function material and
      a rendered-byte digest for the bundle's eligible ordinary command name.
- [ ] The generated function contains only a fixed Atsura template, exact
      safely quoted bindings, and lossless `"$@"` forwarding; configuration
      cannot add shell source.
- [ ] After caller-owned activation, an ordinary `gh ...` fixture invocation
      produces only the plan-declared compact JSON value, not the maintainer
      execution envelope.
- [ ] Wrapper execution derives the source spelling from the bundle, validates
      the rendered binding's expected bundle digest and exact `atr` runtime
      identity before source identity assessment, and uses the same fresh plan
      constructor and exact plan digest as direct preview.
- [ ] Bundle- or runtime-digest mismatch, missing adoption, source drift,
      surface absence, option absence, and unsupported runtime start zero source
      processes; admitted success starts the exact physical source once.
- [ ] Spaces, empty values, Unicode, dash-prefixed values, literal `$()`, and
      argument order survive the generated function and wrapper entry point.
- [ ] Wrapper failure never selects raw execution, another bundle, the ambient
      source name, or a runtime-discovered processor.
- [ ] Production code, catalog, capability ledger, schemas, help, and release
      archives contain no coding-agent-host protocol, settings, permission,
      session, transcript, or model field.
- [ ] A generic caller-owned POSIX fixture activates the generated bytes and
      compares exact argv, plan, result, and attempt evidence; no vendor fixture
      is part of Atsura's compatibility claim.
- [ ] Focused tests, `task check`, `task security`, `task public:check`, and
      `task release:check` pass on the same committed tree; exact installed
      artifact replay covers Linux and macOS POSIX wrapper execution before it
      is described as release quality. Windows remains a regression target for
      existing commands but is not claimed for POSIX activation.
