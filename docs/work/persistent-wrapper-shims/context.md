# Context: Persistent wrapper shims

## Verified facts

- `wrapper render` already proves exact adoption, bundle/source/processor/runtime identity, complete-surface admission, deterministic help, and zero process attempts.
- `wrapper run` already owns fresh-plan parity, no-shell execution, bounded attempts, and exact source/processor delivery.
- Activation is caller-owned; Atsura has no host adapter and must not edit command-resolution configuration.
- The user-local trust store demonstrates confined create-exclusive staging, rename, directory sync, and mutation reconciliation patterns.

## Constraints

- The store is a fixed user-local Atsura root, not a caller-selected arbitrary write target.
- One active shim per ordinary command; a different existing file is a conflict, never an implicit replacement.
- Artifact references are exact opaque store keys and are consumed unchanged.
- Generated bytes come only from a fixed Atsura POSIX template.

## Unknowns deferred from this iteration

- Safe replacement/update UX and multiple purpose profiles
- Persistent executable support on Windows
- Host-specific activation and manual raw execution
