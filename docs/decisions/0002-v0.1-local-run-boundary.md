# ADR 0002: Close v0.1 around one read-only local tailoring run

- Status: Superseded
- Date: 2026-07-21
- Deciders: Repository maintainer
- Scope: Product, architecture, security, harness, public boundary, and release
- Supersedes: None
- Superseded by: docs/decisions/0005-purpose-specific-surface-and-wrapper.md

ADR 0005 retires this ADR's authorization-centered schema-1 public runtime.
The bounded no-shell process and typed output-transform implementations remain
evidence, but they are not a current public bundle-runtime contract.

## Context

`atr plan preview` proves that strict per-command YAML can deterministically
describe an allow/deny decision, argv rewrite, and substantial output reshape.
It does not prove the central wrapper hypothesis because no source process or
output transformation runs.

The public catalog requires every command to declare one effect. An arbitrary
source command could read, create, or write external state, and schema 1 does
not yet carry mutation target or impact facts. Calling arbitrary source
execution a read would violate the operation contract; implementing all
mutation semantics, hooks, discovery, and extensions in one release boundary
would not provide a finite quality target.

## Decision drivers

- Validate the complete deterministic plan-to-process-to-result path.
- Preserve one declared effect and fail closed before source execution.
- Keep executable/argv and invocation/output transformation separate.
- Bound time, memory, output, retries, and executable drift.
- Give users one locally reproducible result without choosing a vendor CLI or host hook.
- Make completion decidable by tests, one manual fixture, and existing gates.

## Considered options

### Option A: Release preview only

This has the smallest side-effect surface, but it cannot establish that one
plan controls execution or that typed transforms can replace real source
presentation.

### Option B: Execute arbitrary source operations

This is broader immediately, but cannot truthfully declare one catalog effect
or safely handle mutation targets, confirmation, uncertain outcomes, and
replay without first designing those contracts.

### Option C: Support one explicitly declared read-only local run

Require schema-1 `effect: read`, execute an allow plan once without a shell,
and support one bounded JSON select/rename/compact pipeline. Defer mutations,
hooks, discovery, raw mode, and extensions.

## Decision

Choose Option C. The v0.1 release-quality product boundary is an explicitly
selected local YAML policy applied by `atr run` to one declared read-only source
invocation. Atsura starts at most one direct process, performs no automatic
retry, accepts successful JSON object or array output, applies typed
select/rename transformation, and returns a fixed execution envelope. Every
failure suppresses raw stdout and never falls back to raw execution.

This is a quality boundary for a future v0.x prerelease, not authorization to
tag or publish. Hooks, command-surface hiding, source inspection, create/write,
confirmation, raw execution, policy persistence, arbitrary shell, jq, RTK,
plugins, and external transformers remain unsupported.

## Consequences

### Positive

- A maintainer can exercise Atsura end to end with any JSON-producing read-only CLI.
- Preview and execution share policy matching, argv transformation, reason, and output-plan semantics.
- Output transformation becomes real product evidence rather than a planned shape.
- A later hook has one bounded local command to invoke instead of owning policy or execution.

### Negative

- Users must explicitly pass one YAML path and assert that the modeled source operation is read-only.
- v0.1 cannot protect a user who misclassifies a mutating source command as read.
- Successful source output must be JSON and fit fixed complexity and byte bounds.
- Source stderr is structurally escaped rather than reproduced byte-for-byte.
- Executable hashing adds local I/O and cannot defend against a compromised OS or source-created children.

### Risks and mitigations

- **False read declaration:** document that effect is a reviewed assertion, accept only `read`, and defer mutations until target/impact policy exists.
- **Executable replacement:** resolve and hash a regular executable, then revalidate before and after the attempt; suppress success on drift.
- **Unbounded process or output:** enforce a 30-second timeout, 4 MiB stdout, 256 KiB stderr, and one direct start attempt.
- **Hostile JSON:** use a duplicate-aware bounded parser, typed values, explicit field selection, and deterministic renderer.
- **Secret leakage:** retain captures only in memory; do not expose raw failed stdout/stderr or upstream causes.
- **Replay after failure:** Atsura never retries; retryability applies only to a later caller decision for the declared read effect.

## Mechanical enforcement

- Domain validation requires `operation.EffectRead` in every schema-1 policy and plan.
- Application fake ports prove all invalid/denied/mismatched inputs make zero process calls and every post-start path has one attempt.
- Process adapter tests cover exact argv, no stdin/shell, timeout, cancellation, byte bounds, nonzero exit, and identity drift.
- JSON adapter/domain tests cover object/array, empty arrays, null/empty/zero/false, duplicate keys, missing fields, depth/node/field/record limits, and deterministic field order.
- Catalog, help, output-schema, hostile-stderr, writer-failure, capability-ledger, and agent-readiness tests cover the public boundary.
- Completion requires full, security, public, and release profiles on one committed tree.

## Compatibility and migration

The repository is unreleased, so schema 1 gains a required `effect: read`
before its first supported prerelease. Existing local preview fixtures must add
that field. v0.x command and schema compatibility is guaranteed within a patch
series; a later v0 minor may change them only with explicit migration notes.
No persisted state exists.

## Security and public-boundary impact

The new side effect is one local source process which may make source-owned
network reads with its inherited environment and credentials. Atsura does not
acquire, store, log, or interpret those credentials. It adds no external
destination or third-party runtime module. Public fixtures are synthetic.


Also run the documented local synthetic source through both `plan preview` and
`run`, observing zero and one direct source-process attempts respectively.

## Reconsideration signals

Create a superseding ADR when the first useful workflow requires create/write,
confirmation, non-JSON output, streaming beyond the fixed bounds, persisted
policy trust, hook interception, raw execution, or an external transformer.
